1.1.x — Foundation: Terminology, Tags, Combatants 2.0, The Workshop (Combatants)

The prerequisite release. Everything in 1.2.x and beyond builds on this
schema and these systems. Nothing in 1.2.x starts until this ships.

  Refactoring
    Terminology — do this first, before any new code is written
      Retire "battle" as a technical term
      Retire "tournament" as a technical term
      Codify the full developer glossary: round → game → series → season → league
        Full definitions, end-condition matrix, resume behaviour, and authority
        model are in docs/glossary.md. That file is the reference — keep it in
        sync as layers are built.
      Update all internal variable names, comments, and UI copy to match
      Audit all existing UI copy and variable names against the glossary
        Confirm no "season" or "league" language appears prematurely in UI strings
        Confirm "battle" and "tournament" are fully retired from all surfaces
      Retire "Bestiary" from all UI copy, variable names, function names, and comments.
        Data and underlying queries unchanged — naming only.
      Retire "Battle History" from all UI copy, variable names, and comments.
      Retire "The Forge" from all UI copy, variable names, and comments.
      File renames:
        BestiaryScreen.jsx → CastScreen.jsx (or ArchiveScreen.jsx with Cast as
          default tab — resolve during implementation; see Archive structure in 1.2.x)
        HistoryScreen.jsx → ChroniclesScreen.jsx
      Update HelpModal.jsx: "Battle history & series" → "The Chronicles",
        "The Bestiary" → "The Cast"
      Update home screen nav buttons: "Battle history" → "The Chronicles",
        "Bestiary" → "The Archive"
      Update user-guide.txt: all references to Battle History, Bestiary, and
        The Forge updated to canonical names
      The Fight Card: add flavor name to draft screen header copy. No structural
        change. Component name and internal references unchanged.

  Schema Migration (prerequisite — nothing else in this release ships without it)
    combatants table
      Add `source text` — 'game' | 'created', default 'game' for all existing rows
      Add `status text` — 'stashed' | 'published'
      Migrate existing `published` boolean:
        published=true  → status='published'
        published=false → status='stashed'
      Drop `published` column after migration
      Add `tags text[]`
      Add `mvp_record jsonb` — array of { gameCode, voteShare, coMvp } entries.
        Replaces the previously planned mvp_votes int — the array gives us the
        count plus per-game narrative detail. Default empty array for all existing rows.
      Populate bio_history on bio edits (the column exists; it's not being written to)
    New tables
      groups(id, name, description, owner_id, owner_name, status, tags, created_at, updated_at)
      combatant_groups(combatant_id, group_id, added_at, added_by)
      arenas(id, name, bio, rules, owner_id, owner_name, status, tags, created_at, updated_at)
        Schema only — no arena feature work in this release. Created here so foreign
        keys and joins are available when arena work begins in 1.2.x.

  Tags
    Free-form text. Lowercase, no special characters, spaces allowed.
    Applied at creation or edit time. Not required — no friction for quick games.
    Displayed as chips/badges on cards and detail pages.
    Autocomplete suggestions with free-form fallback — keeps tags coherent without
      blocking creativity.
    Tags apply to combatants, arenas, and groups. Independent tag spaces per entity
      type for now. Revisit unified cross-entity search if that becomes wanted.
    Super Host admin capability
      Merge duplicate tags after the fact — "spooky" + "spoooky" reconciled and
      applied to all affected records. Super Host role is defined in 1.2.x but the
      merge operation is built here alongside the tags system.
    The Cast becomes searchable and filterable by combatant tag once this ships.

  Combatants 2.0
    Bio history
      Every bio edit appends the previous bio to bio_history with a timestamp and the
        userId/name of who made the change.
      Collapsible "bio log" on combatant detail page — reverse-chronological, no diffs,
        no rollback UI. Just the record. A combatant whose bio has mutated three times
        across games is itself a story. The log IS the narrative artifact.
    Ephemeral / topical tags (system-derived, not user-applied)
      On fire — won 3+ rounds in a row. Badge shows count if the streak extends (4+, 5+, ...).
      Cold streak — lost 3+ in a row.
      Trapper — flag for successful trap combatant record.
      These are derived from match history. Not stored. Not permanent.
    Achievement superlatives
      Not displayed on the small card. Visible on combatant detail page only.
      Full list: things like "beat X opponents", "survived Y games", etc.
      Specific superlatives defined once the stat system has enough data to draw from.
    MVP record
      mvp_record jsonb added in schema migration above. Collect the data now.
      Voting UI and The Cast display ship in 1.3.x.

  The Workshop — Phase 1: Combatants
    Logged-in users only. Guests who try to access see a single-tap login prompt — not
      a wall. Show the benefit, don't lecture.

    The Stash
      Default state for everything created in The Workshop.
      Stashed items are only visible to the owner: in The Workshop, in their profile,
        and in their own draft autocomplete. No other user or guest can see them.
      Stashed items can be freely edited and deleted.
      Once published, deletion is not allowed — it's part of the record.
      Publishing is reversible (un-publish back to stash) but doesn't erase history.
        If the item was ever used in a completed game, it remains accessible through
        that game's record regardless of stash status.

    Create-a-Combatant
      Name (required), bio (required — even a single line), tags (optional),
        group memberships (optional — assign to own or published groups),
        stash/publish toggle.

    My Workshop
      All combatants the user has created — stashed and published.
      Filter tabs: All / Stashed / Published.
      Quick actions: Edit, Publish, Un-publish, Delete (stashed only).
      Creation flow leads directly into the library — same screen, two sides.

    Edit flow
      Bio edits append the previous bio to bio_history.
      Stashed: free editing, no confirmation.
      Published: one-step confirm on bio save — it's now public.

    Draft autocomplete integration
      Logged-in user's stashed combatants appear in their own autocomplete suggestions.
      Visual indicator (lock icon or "stashed" label) distinguishes them from the
        published pool.
      Other players in the same draft do not see these suggestions.

    Publish-on-game-completion
      When a stashed combatant appears in a completed game, it auto-publishes.
      A completed game is a permanent record. A combatant that fought is now part
        of the story — keeping it stashed would create a gap in The Cast.
      Publish happens at room end state, not at draft submit. Draft is reversible;
        game end is not.
      If a game is abandoned before completion, stashed combatants revert to stash.
        No auto-deletion. Stash is the safe holding state.

1.2.x — Game Objects: Arenas, Groups, Super Hosts, The Archive, Open Lobbies

Builds on the schema shipped in 1.1.x. Arenas are gated on a scope decision
that must be made before implementation begins — don't start arena work until
that decision is locked.

  Scope decision required before arenas work begins
    A. Is an arena set at the game level (one arena for the whole session) or at
       the round level (each round can have its own arena)?
    B. Can both be true — a default arena for the game, overrideable per round?
    This decision drives the round and room schema. Resolve it in a design session.
    All arena feature work and The Workshop: Create-an-Arena are blocked until it's resolved.

  Arenas
    Core concept
      An arena is an optional context for a fight: name, description (the setting,
        the vibe, the absurdity), and optional house rules in free text. Not enforced
        by the app — narrative flavour and a prompt for the players.
    Story record
      Arena data is denormalized into the round or room at the time it's selected.
        If the arena is later edited, the fight record still shows what the arena was
        when they fought.
    Likes and dislikes
      Players can like or dislike an arena after playing in it. Running count visible
        on the arena detail page.
      All arenas remain permanently selectable. Enough dislikes relative to likes
        (threshold TBD) soft-removes the arena from curated preset pools. The record
        stays. No arena is deleted.
    Preset pools (for random selection)
      standard — general-purpose arenas
      wacky — high chaos / absurdist
      league — arenas suited for a series home venue (think: a recurring location)
      weighted-liked — community-driven; arenas that accumulate enough dislikes are
        suppressed from this pool automatically
      Presets are compiled through play and curated by Super Hosts.
    Lobby options
      Host can select a specific arena or trigger random from a chosen preset pool.
      One arena for the whole game, or random per round — pending scope decision above.
      Per-round random: host picks the pool; each round draws from it.
      Previous arena from the same series can optionally be excluded to avoid repetition.
    Series behaviour
      An arena used in a series game can persist and appear in follow-up battles.
        It is not available for random selection in a fresh unrelated game.
    The "Tim's house but it's Christmas" case — deferred
      Editing an arena's description or rules for a one-off session without permanently
        altering the original requires a decision: fork, one-session override, or a
        separate "this session only" field. Don't build until resolved.
    The Workshop: Create-an-Arena (gated on scope decision above)
      Name, bio, house rules, tags, stash/publish toggle.
      My Workshop entry — same stash/publish/edit/delete patterns as combatants.
      Publish-on-game-completion applies: a stashed arena used in a completed game
        auto-publishes on room close.

  Groups
    Concept
      A named collective that combatants can belong to. Zero or multiple groups per
        combatant. Examples: "Clown School", "Local Cryptids", "Cursed Object Hall
        of Fame". Groups accumulate history across games, not just within one.
    Why it matters for the story
      When two members of the same group face off, that's a narrative event — a civil
        war, a betrayal, a family feud. The app surfaces it in the round view and ticker.
      Groups give The Cast a new axis: browse by group, see dominant members,
        see group win records (derived from member history, not stored separately).
    Group lifecycle
      Follows the same stash/publish rules as combatants and arenas.
      Either the combatant owner or the group owner can create the membership link.
      When a combatant evolves, the variant inherits group memberships — the variant
        is the same character continued.
      Stashed group memberships are invisible to anyone but the owner. A combatant's
        group list only shows published groups to others.
    Display
      Group badges on combatant cards and detail pages.
      Round view: "civil war" or "group rivalry" flag when both combatants share a group.
      The Cast group view: members, combined record, most decorated member.
    The Workshop: Create-a-Group
      Name (required), description (required — what's the joke, what's the theme),
        tags (optional), stash/publish toggle.
      Member management: add/remove combatants (own or published).
      Group detail page in The Cast: members, combined record.
    Cleanup on delete
      When a stashed combatant is deleted from The Workshop, all combatant_groups rows
        for that combatant are removed.

  Super Hosts
    A trusted user role for game and narrative-level changes made outside of battles.
    Assigned by app admin. Global role — not per-room.
    Powers
      Apply, remove, and edit tags on any published combatant, arena, or group.
      Assign and remove combatants from any group.
      Build and curate arena presets.
      Merge duplicate tags across all affected records.
    What Super Hosts are not
      Not a moderation role — no flagging, no approval flows.
      Not a host override — they cannot control games in progress or alter round outcomes.
      The host still runs the game. Super Hosts curate the world outside it.

  The Archive
    Tabbed destination screen. The Cast is the default tab — users arriving at
      The Archive land on The Cast. Content and behavior identical to current
      Bestiary/Cast; only location and name change.
    Tabs: The Cast (default), Groups, Arenas, Tags
    Unified search bar at Archive level — searches across all entity types
      simultaneously. Returns combatants, groups, arenas, and tags in a single
      results view.
    Tag filtering applies across all tabs — selecting a tag on the Tags tab
      filters The Cast, Groups, and Arenas to matching entries.
    Groups tab: named collectives, member list, combined win/loss record, most
      decorated member. Browsable and searchable.
    Arenas tab: all published arenas, filterable by tag and preset pool.
      Likes/dislikes visible. House rules shown on detail page.
    Tags tab: browse all tags across entity types. Tap a tag to filter across
      the full Archive.
    Home screen nav updated: "The Archive" button replaces the current "Bestiary"
      button. "The Chronicles" button replaces "Battle history". "The Workshop"
      nav item added — replaces current Forge entry point.

  The Workshop renaming
    All remaining Forge references in UI copy updated to The Workshop.
    My Workshop: the expanded library now shows combatants, arenas, and groups —
      the UI label "My Workshop" takes its final form here when all three entity
      types are present.
    "Created in The Forge" → "Made in The Workshop" in any Cast attribution display.
    Stash behaviour, publish flow, and all mechanics unchanged — naming only.

  Open Lobbies
    Option at lobby creation: make the lobby public.
    Public lobby browser in the join flow: sortable list, default sorted by times
      played with host.
    Closed lobbies: explicitly private (current default behaviour, now formally named).
    Joining a public lobby follows the same flow as joining via invite code.

1.3.x — Season

Builds on the clean terminology and series concept from 1.1.x. Season is the
layer above series: a structured container with a fixed player roster, declared
series count, cumulative standings, and end-of-season awards. Requires no
heritage carryover between series — each series starts fresh.

  Schema
    seasons table
      id, name, league_id (nullable — supports standalone seasons), owner_id,
      owner_name, status ('active' | 'ended' | 'cancelled'),
      series_count (declared target), series_played (running count),
      latest_evolutions_only (boolean — stored now, enforcement deferred),
      created_at, updated_at

  Features
    Season creation flow
      Name, player roster (explicit opt-in), declared series count,
      "latest evolutions only" toggle (stored at creation; enforcement is
      deferred — see glossary for scope).
    Season standings page
      Cumulative round wins / losses / draws per player across all series
      in the season.
    Season close logic
      Auto-close when series_played reaches series_count at the end of a series.
      Manual early-close available to season creator between series only.
      A season cannot be closed mid-series.
    Season resume
      Creates a new fresh series under the same season_id.
      series_played increments. No prev winners or heritage carryover.
    Awards — fires on season close (both natural and early)
      Automated awards — computed immediately on close:
        most round wins
        most traps set
        most efficient trapper (minimum trap threshold TBD once data exists)
        most evolutions triggered
      Voted awards — ballot opens for season members on close:
        favorite combatant (season-scoped, distinct from per-game MVP vote)
        most creative combatant (season-scoped)
      Awards should feel like chapter titles, not leaderboard ranks.
    MVP voting
      Optional flow on the game end screen. First nomination opens the process —
        once open it cannot be cancelled, only closed early by the host.
      Nomination pool: round winners shown by default; full combatant dropdown
        available as secondary option for memorable losers.
      Secret ballot — picks hidden until resolution. Live status strip shows who
        has locked in a nomination or abstained (names visible, picks hidden).
        Mirrors ConnectionStatus pattern.
      Resolution paths:
        1. Clear winner — auto-awarded, no prompt.
        2. Tie after all players lock in — tiebreak runoff between tied nominees.
           Runoff resolves cleanly → MVP awarded.
           Runoff deadlocks, or not all players lock in before host closes →
             co-MVP, auto-resolved, no further prompts.
        3. Host closes early at any point — resolve with current votes; ties
             auto-resolve as co-MVP.
      On resolution: append to mvp_record on each winning combatant —
        { gameCode, voteShare, coMvp }.
      Co-MVP: each tied combatant receives a separate entry, coMvp: true,
        voteShare split evenly.
      The Cast combatant detail: MVP history — "MVP in ROOM X — 67% of the vote"
        per entry, derived from mvp_record.
      MVP count (mvp_record.length) feeds into season-level automated awards.
      mvp_record is populated from 1.1.x — data is ready when this UI ships.
    Arena integration
      A season can designate a home arena — a recurring venue for the run.
      Optional: sub-arenas within a season realm. Build only if the base arena
        system warrants it after 1.2.x ships.

  Deferred
    "Latest evolutions only" enforcement
      Store the setting at season creation. Implement draft-time lock in a
      follow-up once complexity is fully assessed. Do not block season launch
      on this.

1.4.x — League

Builds on the season layer from 1.3.x. League is the top tier: a persistent
named group that runs multiple seasons, with cumulative standings across all
of them. Each new season starts fresh — no heritage carryover between seasons.

  Schema
    leagues table
      id, name, owner_id, owner_name, status ('active' | 'ended' | 'cancelled'),
      season_count (declared target), seasons_played (running count),
      created_at, updated_at

  Features
    League creation flow
      Name, player roster (explicit opt-in), declared season count.
    League standings page
      Cumulative stats and notable moments across all seasons in the league.
    League close logic
      Auto-close when seasons_played reaches season_count at the end of a season.
      Manual early-close available to league creator between seasons only.
      A league cannot be closed mid-season.
    League resume
      Creates a new fresh season under the same league. No carryover.
      seasons_played increments.
    Awards
      Same automated + voted award pattern as season, but league-scoped.
      Fires on league close.

1.5.x — Users Lounge

A separate layer built on top of the compendium created in 1.1.x–1.4.x. The
items here are a vision, not a spec. A dedicated design doc is needed before
any of this is planned for implementation.

  Features
    User decks
      Combatants released periodically as collectible trading cards, exclusive to
        the Users Lounge. No effect on main game stats or records.
      Cards are 1 of 1.
        Open question: how does uniqueness work when a combatant appears in multiple
          games? Is a new card minted per game appearance, per evolution, per season?
          Must be resolved before building.
      Draft: users draft to select newly released cards for their decks.
      Trading: cards can be traded between users.
      Gambling: blind or known wagering for cards.
        Needs its own design pass — raises questions about reversibility, dispute
          handling, and UX when a wager goes sour.
    Minigames
      Narrative games (history not kept by default — worth revisiting per game)
        Airplane — lay x cards in rows, flip one at a time. No substance. Good for a laugh.
        Football — 1v1, each user puts cards in a 7-on-7 grid. Flip asynchronously
          to build the narrative of the play.
        More TBD.
      More substantial games
        8s with your deck — select 8 cards from your deck (random draw or manual select)
          and run a game. Could be recorded as a minigame stat.
    Open to other ideas.
