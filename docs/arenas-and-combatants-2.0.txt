ARENAS + COMBATANTS 2.0
Brainstorm / Design Document
Last updated: 2026-04-13
Status: Active brainstorm — not yet scheduled for implementation

---

OVERVIEW

This document captures the planned enhancements to combatants and arenas as
standalone concepts — before "The Forge" (user creation feature) is built on
top of them. The Forge depends on most of what's described here being in place
first. This doc is intentionally forward-looking; not every idea here is
confirmed for build.

The guiding question for every addition: does this make the story richer, or
does it add noise?

---

PART 1: COMBATANT BIOS AS A LIVING DOCUMENT

Current state:
  - `bio_history` is a jsonb field on the combatants table
  - It's not currently populated — evolution records the shape but bio edits
    don't write to it

What we want:
  - Every time a combatant's bio is changed, the previous bio is appended to
    bio_history with a timestamp and the userId who made the change
  - A lightweight "bio log" visible on the combatant detail page: collapsible,
    showing previous bios in reverse-chronological order
  - No diffing, no rollback UI — just "here's what it used to say, and when
    it changed"

Entry shape in bio_history:
  {
    bio: "the text as it was",
    changedAt: "2025-11-02T14:32:00Z",
    changedBy: "userId",         // who made the edit that replaced this bio
    changedByName: "username"    // denormalized for display
  }

Is a changelog overkill?
  Not for this app. The whole point is that years from now the record still
  tells the story. A combatant whose bio has mutated three times across
  tournaments is itself a story. The changelog IS the narrative artifact.
  Keep it simple — no diffs, no version numbers, just the old bios in order.

PART 2: TAGS

Both combatants and arenas should support optional free-form tags. Tags serve
two purposes: browsing/filtering in the Bestiary and The Forge, and eventual
matchup-flavour opportunities (two "Clown School" members fighting, etc.).

Tag design principles:
  - Free text, lowercase, no special characters (spaces allowed)
  - Applied by the creator at creation or edit time
  - Not required — no friction for quick creation
  - Displayed as small chips/badges on cards and detail pages

Combatant tags (examples to guide tone, not prescriptive):
  - Style: "undead", "animal", "mythical", "professional", "inanimate"
  - Tone: "absurdist", "wholesome", "cursed", "chaotic"
  - Origin: "household", "internet", "historical", "local legend"

Arena tags (examples):
  - Setting: "indoor", "outdoor", "fantasy", "mundane", "domestic"
  - Mood: "ceremonial", "chaotic", "cramped", "grand"
  - Tone: "absurdist", "dramatic", "comedic"

Schema:
  - `tags text[]` column added to `combatants` and `arenas`
  - No separate tags table needed at this point
  - If we ever want tag-level stats or descriptions, revisit

Open questions:
  - Curated tag suggestions vs completely free-form? Suggestion list with
    free-form fallback seems right — keeps tags coherent without blocking
    creativity.
  - Should tags be filterable in the Bestiary before The Forge exists? Low
    priority but nice. Table for now.

PART 3: GROUPS / GUILDS / GANGS

This is the most narrative-rich addition in this document.

Concept:
  A Group is a named collective that combatants can belong to. Think: "Clown
  School", "The Wednesday Regulars", "Cursed Object Hall of Fame", "Local
  Cryptids". A combatant can belong to zero or multiple groups.

Why this matters for the story:
  - When two members of the same group face off in a round, that's a narrative
    event — a civil war, a betrayal, a family feud. The app should recognise it
    and surface it.
  - Groups give the Bestiary a new axis: you can browse by group, see which
    groups have the most wins, who the dominant member of a group is.
  - Groups created by players accumulate history across tournaments, not just
    within one game.

Group schema (new table):
  groups(
    id          uuid PK
    name        text
    description text          -- what this group is / the joke behind it
    owner_id    uuid FK users
    owner_name  text          -- denormalized
    status      text          -- 'stashed' | 'published'
    tags        text[]        -- optional, same system as combatants/arenas
    created_at  timestamptz
    updated_at  timestamptz
  )

  combatant_groups(           -- many-to-many junction
    combatant_id  uuid FK combatants
    group_id      uuid FK groups
    added_at      timestamptz
    added_by      uuid FK users
  )

Group lifecycle:
  - Groups follow the same stash/publish rules as combatants and arenas
  - A stashed group is only visible to the owner
  - A published group is visible to all and browsable in the Bestiary
  - Combatants can be added to groups at creation time or later from their
    detail page / The Forge

Group affiliation display:
  - Combatant cards and detail pages show group badges
  - Round view: if both combatants share a group, surface a "civil war" or
    "group rivalry" flag in the round narrative / ticker
  - Bestiary group view: members, win record, most decorated member

Open questions:
  - Can a group have its own win/loss record (aggregate of all member fights)?
    Yes, probably — but derive it, don't store it. A group's "record" is
    computed from the win/loss of all members in rounds where both appeared.
    Actually simpler: just the aggregate of all member wins/losses.
  - Who can add a combatant to a group? The combatant's owner, or the group's
    owner, or both? Start with: either the combatant owner or the group owner
    can create the link.
  - Can a group be "locked" so only the owner can add members? Table for now.

PART 4: ARENAS

Tabled for full implementation but captured here for continuity.

Core concept:
  An arena is an optional context for a fight. It has a name, a description
  (the setting, the vibe, the absurdity), and optional house rules in free text.
  It's not enforced by the app — it's narrative flavour, a prompt for the
  players, part of the story record.

Two scope questions that must be answered before building:
  A. Is an arena set at the tournament level (one arena for the whole game)
     or at the round level (each round can have its own arena)?
  B. Can both be true? (default arena for the tournament, overrideable per round)

Pending decision: defer until arena brainstorm session.

Arena schema (captured for reference):
  arenas(
    id          uuid PK
    name        text
    bio         text          -- setting, flavour, description
    rules       text          -- optional house rules (free text)
    owner_id    uuid FK users
    owner_name  text
    status      text          -- 'stashed' | 'published'
    tags        text[]
    created_at  timestamptz
    updated_at  timestamptz
  )

  groups could optionally have an associated "home arena" — a fun detail but
  non-essential.

Random arena picker:
  - Pool: all published arenas + the host's own stashed arenas
  - Can be triggered at room creation or at round start (depending on scope
    decision above)
  - Previous arena from the same series could be excluded to avoid repetition
    (optional, not essential)

Arena in the story record:
  - Once a round or tournament has an arena, it's permanent. The arena name and
    bio are stored with the round/room data at time of fight, not as a live
    reference. If the arena is later edited or deleted, the fight record still
    shows what the arena was when they fought.
  - This means: denormalize arena data into the round or room object at the
    time it's selected.

PART 5: SCHEMA MIGRATION NOTES

Changes required before The Forge can be built:

combatants table:
  - Add `source text` — 'game' | 'created', default 'game' for all existing rows
  - Add `status text` — 'stashed' | 'published'
  - Migrate existing `published` boolean:
      published=true  → status='published'
      published=false → status='stashed'
  - Drop `published` column after migration
  - Add `tags text[]`
  - Ensure `bio_history` is being populated on bio edits (it's not currently)

New tables:
  - `arenas` (when arena work begins)
  - `groups`
  - `combatant_groups`

Note on RLS:
  The app uses PIN-based auth rather than Supabase Auth tokens, so
  `auth.uid()` RLS policies won't work for stash filtering. Visibility
  enforcement is at the query layer: stashed rows are only fetched when
  querying with the owner's userId. This is acceptable for a trusted
  friend-group app. Don't build RLS policies that depend on Supabase Auth
  until/unless the auth model changes.

OPEN QUESTIONS (parking lot)

  - Should tags be shared across combatants, arenas, and groups (one canonical
    tag list), or is each entity's tag space independent? Independent for now
    is simpler; unified later if browsing by tag across entity types is wanted.

  - Should groups appear in the Bestiary as first-class browsable entities,
    alongside combatants? Yes, eventually. Not in the first pass.

  - Evolution and groups: if a combatant evolves into a variant, does the
    variant inherit group memberships? Probably yes — the variant is the same
    "character" continued. Worth confirming.

  - Can arenas belong to a group? ("Clown School has a home arena: The Big Top")
    Fun idea, low priority. Note it and move on.
