STEP 1 — Fix guestId stability (prerequisite for everything else)
================================================================================

Problem:
  guestId lives in sessionStorage (App.jsx:73). It resets on tab close, hard
  refresh, or "open in new tab." Lobby codes live in localStorage, so after any
  session break the guest can see their room codes but no longer matches the
  player record in any of them. This is the root cause of Ticket 5.

Fix:
  File: src/App.jsx, lines 72–76

  Change:
    const s = sessionStorage.getItem('eights_pid') || uid()
    sessionStorage.setItem('eights_pid', s)

  To:
    const s = localStorage.getItem('eights_pid') || uid()
    localStorage.setItem('eights_pid', s)

  That's it. Keep the useState wrapper. The guestId is now stable across
  refreshes, tab close, and browser restart. Only lost on explicit storage clear.

Note: playerName stays in sessionStorage (eights_pname) — intentional, it's
  per-session. Only the identity token moves to localStorage.

STEP 2 — Extend trackRoomPresence to expose all participants
================================================================================

Problem:
  trackRoomPresence (supabase.js:98) currently calls back with a single boolean
  (host online/offline). The host can't see who else is connected.

File: src/supabase.js

Change the function signature from:
  trackRoomPresence(roomId, userId, role, onHostStatusChange)

To:
  trackRoomPresence(roomId, userId, role, { onHostStatusChange, onPresenceChange })

  where onPresenceChange(presentIds: string[]) is called on every presence sync.

Updated implementation:

  export function trackRoomPresence(roomId, userId, role, callbacks = {}) {
    const { onHostStatusChange, onPresenceChange } = typeof callbacks === 'function'
      ? { onHostStatusChange: callbacks }   // backwards-compat: old callers pass fn directly
      : callbacks

    let lastKnown = null
    const channel = supabase
      .channel('presence-' + roomId, { config: { presence: { key: userId } } })
      .on('presence', { event: 'sync' }, () => {
        const state = channel.presenceState()
        const flat = Object.values(state).flat()
        const hostOnline = flat.some(p => p.role === 'host')
        if (hostOnline !== lastKnown) {
          lastKnown = hostOnline
          onHostStatusChange?.(hostOnline)
        }
        onPresenceChange?.(flat.map(p => p.key ?? p.userId).filter(Boolean))
      })
      .subscribe(async (status) => {
        if (status === 'SUBSCRIBED') {
          await channel.track({ role, userId })
        }
      })
    return () => supabase.removeChannel(channel)
  }

Note: the backwards-compat branch (typeof callbacks === 'function') means existing
  callers in BattleScreen.jsx:38 and VoteScreen.jsx:48 don't need to change until
  they want the new data. Update them in Step 3.

Presence key: Supabase presence uses the channel key (userId) to identify slots.
  Add userId to the tracked payload so flat.map can retrieve it regardless of key
  collisions (unlikely but safe). The channel config already sets key: userId.

STEP 3 — Connection status block (always-visible presence indicator)
================================================================================

Design:
  A small "who's connected" strip that appears on every in-room screen — lobby,
  draft, battle, vote. Shows all players as colored dots with their name. Dot is
  solid = connected, hollow/dim = disconnected. Host always sees it; non-hosts
  see it too (useful for remote play: "why hasn't Alice voted?").

  For the HOST only: tapping a disconnected player opens a small action sheet:
    - [Guest player] "Share rejoin link" — copies/shows a URL with ?code=XXXX&pid=PLAYERID
    - [Any player]   "Remove from room"  — existing admin escape hatch if applicable
  Tapping a connected player does nothing (or shows their name/color as confirmation).

Component: src/components/ConnectionStatus.jsx  (new file)

Props:
  players        — room.players (full list)
  presentIds     — string[] from onPresenceChange callback
  isHost         — bool
  roomCode       — string (for link generation)
  onShareLink    — fn(player) — called when host taps a disconnected guest

Rendering:
  A horizontal strip of small avatar/dot pairs. Fits in a fixed strip below the
  screen header or above the main content area. Keep it compact — 28px height max.
  Use existing playerColor(index) for dot color (imported from gameLogic.js).
  Disconnected state: opacity 0.35, ring instead of filled dot.

  Example layout (host view, Alice online, Bob offline guest):
    ● Alice  ○ Bob (guest)
  Bob's row is tappable for the host.

Rejoin link format:
  https://[domain]/?code=XXXX&pid=PLAYERID

  Generate with: window.location.origin + '/?code=' + roomCode + '&pid=' + player.id

  The URL is copyable via navigator.clipboard.writeText(). Show a brief "Copied!"
  confirmation inline — no modal needed.

Consuming the ?pid param in App.jsx:
  On mount, before the useState for guestId initializes, check:
    const urlParams = new URLSearchParams(window.location.search)
    const urlPid = urlParams.get('pid')
    const urlCode = urlParams.get('code')

  If urlPid is present, write it to localStorage as 'eights_pid' before the
  guestId useState reads it. The guest then loads with their old identity.
  If urlCode is also present, pre-fill it on the join screen (or auto-join).

  Implementation note: do this outside of useState (run it synchronously at
  module level or in the useState initializer function). The guestId useState
  already uses a function initializer — add the URL check there:

    const [guestId] = useState(() => {
      const params = new URLSearchParams(window.location.search)
      const urlPid = params.get('pid')
      if (urlPid) {
        localStorage.setItem('eights_pid', urlPid)
        // Clean the URL so the param doesn't persist on refresh
        window.history.replaceState({}, '', window.location.pathname + (params.get('code') ? '?code=' + params.get('code') : ''))
        return urlPid
      }
      const s = localStorage.getItem('eights_pid') || uid()
      localStorage.setItem('eights_pid', s)
      return s
    })

  Also handle ?code= auto-fill: after the component mounts, if urlCode is set,
  navigate to the join screen with the code pre-filled. The cleanest place is
  a useEffect in App.jsx that fires once on mount.

Where to mount ConnectionStatus:
  - LobbyScreen.jsx: after the player list header, before the player rows
  - BattleScreen.jsx: below the screen title, above the round list
  - VoteScreen.jsx: same position as BattleScreen
  - DraftScreen.jsx: optional — presence less critical during draft, but low cost
    to include

  Each screen already calls trackRoomPresence (BattleScreen, VoteScreen) or will
  need to add it (LobbyScreen — currently has no presence tracking). Pass the
  presentIds from onPresenceChange into ConnectionStatus.

  For LobbyScreen, add trackRoomPresence call in a useEffect (same pattern as
  BattleScreen.jsx:37–39). LobbyScreen doesn't need onHostStatusChange; just
  onPresenceChange.

Is player a guest?
  The room.players array has { id, name, color, ready }. There is no isGuest flag
  stored on the player record — it's inferred from whether the id matches a user
  account. For the rejoin link, it doesn't matter: generate the link for ANY
  disconnected player. If they're logged in, they don't need it; if they're a
  guest, they do. The host doesn't need to know the distinction to offer the link.

  Optional enhancement: show "(guest)" next to a player name in the strip if their
  id is not in the registered users list. This requires an async lookup. Defer this
  — the host can tell by context who's a guest. Don't block the feature on it.

STEP 4 — Host-guest warning at CreateRoom (highest-value nudge)
================================================================================

Why this is Priority 1:
  The principles-audit.txt Priority 1 identifies three guest nudge moments.
  The CreateRoom moment is the most critical: the HOST losing their session means
  no one can advance the game. This is not a "save your history" nudge — it is
  a "you are the only person who can run this room" warning.

File: src/screens/CreateRoom.jsx

Props available: CreateRoom receives no currentUser or isGuest prop today.
  Add: isGuest (bool) passed from App.jsx where it's already computed.

App.jsx already passes currentUser to many screens. Add isGuest={isGuest} to
  the CreateRoom render call in App.jsx.

Warning placement: below the name field (if lockedName is false, i.e., guest),
  above the Settings section. Appears only when isGuest is true.

Copy (matches the voice — not lecturing, just making the consequence clear):
  "You're creating as a guest. If you close this tab or switch devices, you'll
  lose host controls and the game can't continue. Log in first to keep your seat."
  [Log in →]

Visual treatment: same inline style as the existing locked-name note at line 77
  (fontSize 12, color var(--color-text-tertiary)) but with a warning color for
  emphasis — var(--color-text-warning). One line of text, one inline link.

The [Log in →] link calls onLogin (needs to be threaded through as a prop).
  After login, the user returns to CreateRoom with their name locked (lockedName
  becomes true) and the warning disappears.

  App.jsx already has afterAuth state for post-login navigation. Wire it so
  tapping "Log in" from CreateRoom returns to CreateRoom after auth:
    setAfterAuth(() => () => nav('create'))
    nav('auth')

  This is the same pattern used in JoinRoom.jsx for the existing login nudge.

Second warning — before game start (host is still a guest in the lobby):
  If the host reaches LobbyScreen as a guest and taps "Start game," show a
  one-time confirmation:

    "You're still a guest. Once the game starts, losing your session means
    players can't advance. Start anyway?"
    [Log in first]  [Start anyway]

  "Log in first" navigates to auth with afterAuth returning to lobby.
  "Start anyway" proceeds as normal.

  This is a gate, not a wall — the host CAN start as a guest with one extra tap.
  Implementation: add a `confirmGuestStart` state in LobbyScreen. If isGuest and
  not yet confirmed, show the prompt instead of immediately calling startGame().
  isGuest needs to be passed into LobbyScreen (add to props from App.jsx).

STEP 6 — Admin: Re-attribute guest history to a user account
================================================================================

Use case:
  A guest played, their combatants were published, they later create an account.
  Admin needs to link the old guest combatants and room history to the new account.

How to find the guest ID:
  Use the existing Inspector tab (AdminScreen > Inspector). Look up the room code,
  expand the JSON, find the guest player in room.players[]. Their id is the guestId.
  This is already possible — no new tooling needed for the lookup step.

New admin section: "Re-attribute guest" in src/screens/admin/UsersTab.jsx
  Add below the MergeSection. Same visual pattern (bordered section, h3 header).

UI:
  1. Text field: "Guest ID" — paste the raw ID found in Inspector
  2. Button: "Look up" — fetches all combatants and rooms with this owner_id
  3. Preview: "Found 8 combatants in 2 rooms attributed to guest ID abc1234
     (owner_name: Mike). Assign to:"
  4. User dropdown (same users list as MergeSection)
  5. Confirm button → runs migration

New Supabase function: adminLinkGuestToUser(guestId, userId, ownerName, rooms, applyMergeToRoom)
  File: src/supabase.js

  Steps:
    a. Update combatants table: SET owner_id = userId, owner_name = ownerName WHERE owner_id = guestId
    b. For each room in rooms that contains a player with id === guestId:
         apply replacePlayerIdInRoom(room, guestId, userId)
         sset('room:' + room.id, updatedRoom)
    c. No account deletion (no guest account exists)

  This is a subset of adminMergeUsers — copy the combatants update and room
  migration steps, remove the user deletion step.

New supabase helper to fetch by owner_id (for the preview):
  async function getCombatantsByOwnerId(ownerId) — select from combatants where owner_id = ownerId

Self-service path (optional enhancement, lower priority):
  When a guest logs in for the first time and their guestId has completed rooms
  with published combatants, offer in-app:
    "You played as a guest before. Claim your combatants to this account?"
    [Claim] [Not mine]
  [Claim] runs the same migration as above, silently.
  Trigger: in the login useEffect in App.jsx, after the existing guest-migration
  block (App.jsx:142–154), add a check for completed rooms (phase === 'ended' or
  'completed') with the guestId. If found, set a pendingClaim state to show the
  prompt. Use a modal or banner — don't block the home screen.
